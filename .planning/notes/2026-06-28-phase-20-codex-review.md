# Phase 20 Cross-Model Review (Codex)

## [A] Publish auth - verdict REVISE

Strongest argument: the recommendation correctly identifies the Node/npm OIDC blocker, but it misses a higher-blast-radius npm prerequisite in this repo: the publishable package manifests do not declare a `repository.url`, while current npm trusted-publishing troubleshooting says GitHub publishes require the package `repository.url` to exactly match the GitHub repository. `packages/mcp/package.json:1-34` and `packages/cli/package.json:1-34` have no `repository` field; `package.json:1-36` has no root `repository` either. That should be fixed before the first trusted-publisher tag, and the lockfile updated with the same manifest change.

Repo evidence: `release.yml` requests `id-token: write` at `.github/workflows/release.yml:8-10`, but sets up Node 20 at `.github/workflows/release.yml:21-26` and publishes with no `NODE_AUTH_TOKEN` at `.github/workflows/release.yml:37-41`. Current npm docs say trusted publishing requires npm CLI >= 11.5.1 and Node >= 22.14.0, so the current workflow is expected to fail authentication before any package is published if it tries OIDC. However, the "fails safe, publishes nothing" claim is too broad for this two-package workflow: the workflow publishes `@localground/mcp` first and `@localground/cli` second in separate steps (`.github/workflows/release.yml:37-41`). If one npm trusted-publisher config is correct and the other is not, a partial release is plausible: mcp 3.0.1 published, cli 3.0.1 failed. That is not a broken tarball, but it is still an irreversible bad release state for a paired toolkit.

Other publish-auth findings: trusted publisher can be configured post-hoc on existing packages; npm's docs describe editing trusted-publisher configuration in package settings. Account/package 2FA is not the blocker for OIDC; npm documents that token-disallow settings affect traditional tokens, while trusted publishers continue using OIDC. Token fallback with `NODE_AUTH_TOKEN` plus `--provenance` is real; npm's provenance docs show `id-token: write`, `npm publish --provenance --access public`, and `NODE_AUTH_TOKEN`. I would keep the granular-token fallback as an emergency path, but not make it primary. I could confirm granular tokens have configurable expiration and Bypass 2FA defaults false; I did not find current npm docs confirming the "90-day cap" claim.

Additional repo mechanism: npm recommends no package-manager caching in release builds. The release workflow currently uses `cache: 'npm'` in `.github/workflows/release.yml:21-26`. Remove release-job caching or move to the current npm-recommended setup-node form with package-manager cache disabled.

## [B] Sequencing - verdict REVISE

Strongest argument: incremental release is the right operating model for a first automated release, but the sequence needs one more guardrail: do not tag until the exact commit containing the release workflow fix, manifest metadata fixes, package-lock update, and 3.0.1 versions has already passed CI on master. "Push master as-is, then bump, then tag" is only safe if "as-is" means "the no-release CI discovery push," not "the final tag target."

Repo evidence: CI only runs on `master` push and PRs (`.github/workflows/ci.yml:3-7`), while release runs on any `v*` tag (`.github/workflows/release.yml:3-6`). The release workflow does not check that tag `v3.0.1` matches package versions before publishing. Current versions are 3.0.0 in root/core/mcp/cli manifests (`package.json:1-11`, `packages/core/package.json:1-20`, `packages/mcp/package.json:1-34`, `packages/cli/package.json:1-34`), while `.claude-plugin/plugin.json:1-8` is already 3.0.1. The lockfile is currently aligned with 3.0.0 at `package-lock.json:1-10` and workspace entries `package-lock.json:3160-3203`; local `npm install --package-lock-only --ignore-scripts --dry-run` reported "up to date" for the current state. After bumping, package-lock must be updated in the same commit or `npm ci` becomes the first likely failure point.

Concrete sequence I would use: push current master once to get the first 3-OS CI signal; then commit release workflow fixes plus package metadata and lockfile if needed; push and wait for CI; then commit/bump 3.0.1 across the intended manifests and lockfile; push and wait for CI on that exact commit; create annotated tag `v3.0.1` on that commit; verify with `git show v3.0.1:packages/mcp/package.json` and `git show v3.0.1:packages/cli/package.json` before pushing the tag. Big-bang "fix and tag before any master CI run" is worse for this owner because it combines first GitHub push, first CI, first OIDC, and first immutable npm publish into one failure surface.

## [C] Windows CI - verdict CONFIRM

Strongest argument: the targeted pre-audit is worth doing, but it should stay narrow. I grepped the repo for both known patterns. The specific hosted-Windows short-path round-trip issue appears already fixed where it matters: `packages/core/test/environment/decode.test.ts:10-16` canonicalizes the temp fixture through `fs.realpath()` and documents the `RUNNER~1` versus long-name mismatch. Other tests use `os.tmpdir()` fixtures, but I did not find another encode/decode/readdir round-trip that needs the same realpath treatment.

Repo evidence: the real path-hash round-trip tests live in `packages/core/test/environment/decode.test.ts:53-71` and special-character cases at `packages/core/test/environment/decode.test.ts:110-211`. The decoder mechanism actually walks the filesystem through `fs.readdir()` and compares `encode(entry.name)` (`packages/core/src/environment/decode.ts:154-191`), which is exactly why the `fs.realpath()` fixture fix matters. The broader tmpdir users in copy/chunk/checksum/compare/placeholder/CLI tests are real-fs fixtures, but they are not performing that decoder reverse-lookup invariant.

Spawn audit: outside `scripts/verify-tarball.mjs`, I found no npm/npx spawn that would hit the Windows `.cmd`/Node 20 EINVAL path. The tarball verifier intentionally spawns npm through `process.execPath` plus `npm-cli.js` (`scripts/verify-tarball.mjs:33-63`, `scripts/verify-tarball.mjs:117-145`, `scripts/verify-tarball.mjs:180-201`). CLI/MCP tests spawn the built JS entry through `process.execPath` (`packages/cli/test/smoke.test.ts:36-41`, `packages/cli/test/json-mode.test.ts:26`, `packages/mcp/test/smoke.test.ts:18`, `packages/mcp/test/stdout-discipline.test.ts:19-60`). The remaining `spawnSync` sites are git invocations in tests and the core `spawnTool` wrapper; they are not npm/npx shims.

Other hosted-Windows gotcha: release will build on Node 22 after [A], but CI intentionally gates Node 20 across three OSes (`.github/workflows/ci.yml:23-24`) and tsup targets Node 20 in both publishable packages (`packages/mcp/tsup.config.ts:14`, `packages/cli/tsup.config.ts:14`). That split is acceptable: Node 22 is a publish-auth requirement, not a runtime-floor change.

## [D] Install scope - verdict CONFIRM

Strongest argument: do not add a bundled `.mcp.json` in Phase 20. It is not required by any Phase 20 success criterion, it reverses the Phase 19 runtime decision, and it changes user install semantics by auto-registering/auto-starting a server when the plugin loads. For a first automated release, documented/manual MCP registration plus post-publish validation is the lower-risk path.

Repo evidence: Phase 20 success criteria require npm publish, provenance badge, README rendering, and `npx -y @localground/cli@3.0.1 detect` smoke (`.planning/ROADMAP.md:87-96`), not bundled MCP auto-registration. Phase 19 explicitly keeps npm-distribution CLI smoke in Phase 20, not Phase 19 (`.planning/phases/19-skill-runtime-uat/19-CONTEXT.md:32-34`, `.planning/phases/19-skill-runtime-uat/19-CONTEXT.md:113-117`, `.planning/phases/19-skill-runtime-uat/19-CONTEXT.md:159`). The plugin manifest is only name/description/version (`.claude-plugin/plugin.json:1-8`), and `Get-ChildItem -Recurse -Filter .mcp.json` found no bundled config. The skills require MCP tool names in frontmatter (`skills/seed/SKILL.md:1-6`, `skills/migrate/SKILL.md:1-8`, `skills/reap/SKILL.md:1-7`, `skills/cleanup/SKILL.md:1-6`, `skills/verify/SKILL.md:1-5`), so the server must still be registered separately.

README evidence: the MCP package README gives install commands for macOS/Linux and Windows (`packages/mcp/README.md:7-21`), including the Windows `cmd /c npx` workaround at `packages/mcp/README.md:14-19`. The CLI package README gives the npx command and commands table (`packages/cli/README.md:7-38`). Local `npm pack --dry-run --json -w @localground/mcp` and `-w @localground/cli` both included `README.md`, `package.json`, and `dist/*`, matching DOC-03 packaging expectations; actual npm page rendering remains a post-publish check per `.planning/REQUIREMENTS.md:53-55`.

## Cross-cutting gaps

1. Missing `repository.url` in publishable package manifests is the biggest gap. Without it, GitHub trusted publishing/provenance is likely to fail even after Node/npm are fixed. Add exact GitHub repo metadata to `packages/mcp/package.json` and `packages/cli/package.json`; consider root metadata too for consistency.

2. The release workflow has no version/tag preflight. Add a pre-publish step that asserts `GITHUB_REF_NAME` equals `v$(node -p "require('./packages/mcp/package.json').version")` and the CLI version matches too. This prevents the non-developer failure mode where the tag points at the wrong commit.

3. Sequential two-package publish can partially succeed. Because `@localground/core` is private and bundled via `noExternal` (`packages/mcp/tsup.config.ts:16`, `packages/cli/tsup.config.ts:16`), there is no core publish-order problem. The real order problem is partial mcp/cli publication. A dry-run/package-shape gate before both real publishes reduces artifact risk, but it does not make npm publish transactional.

4. Release-job npm caching conflicts with current npm guidance for release builds. `release.yml` currently caches npm (`.github/workflows/release.yml:21-26`); remove it for the release workflow even if CI keeps cache.

5. Package metadata is thin. The publishable manifests have no `license`, `repository`, `homepage`, or `bugs` fields (`packages/mcp/package.json:1-34`, `packages/cli/package.json:1-34`). Not all are release blockers, but `repository` is auth/provenance-relevant and `license` affects npm page quality.

6. Token fallback claims need tighter wording. Token + provenance is valid, and Bypass 2FA defaults false, but I did not confirm a current npm-docs "90-day cap" on granular tokens. Do not encode that as a release assumption unless the maintainer verifies it in npmjs.com at token creation time.

## Bottom line

Before the maintainer pushes the first release tag, I would revise the plan as follows: keep OIDC as the primary path, but fix `release.yml` to Node 22.x/npm >= 11.5.1, remove release caching, add exact `repository.url` metadata to both publishable packages, update package-lock after any manifest/version changes, and add a tag/version preflight before `npm publish`. Then push master for CI, wait for green, tag only the exact 3.0.1 commit, and manually verify both package pages plus `npx -y @localground/cli@3.0.1 detect` and the README-documented MCP registration command. Do not ship `.mcp.json` in this patch release.
