```yaml
name: Release
on:
  push:
    tags:
      - 'v*'
permissions:
  contents: read
  id-token: write
jobs:
  release:
    name: Build, test, and publish to npm
    runs-on: ubuntu-latest
    steps:
      - name: Checkout repository
        uses: actions/checkout@v4
      - name: Set up Node.js 20.x with npm registry
        uses: actions/setup-node@v4
        with:
          node-version: '20.x'
          registry-url: 'https://registry.npmjs.org'
          cache: 'npm'
      - name: Install dependencies (npm ci)
        run: npm ci
      - name: Build all workspace packages
        run: npm run build
      - name: Run test suite
        run: npm test
      - name: Publish @localground/mcp with provenance
        run: npm publish -w @localground/mcp --provenance --access public
      - name: Publish @localground/cli with provenance
        run: npm publish -w @localground/cli --provenance --access public
```

Reference for the dry-run spawn shape (Windows-safe pattern, FYI — release runs on ubuntu so plain `npm` works in YAML run steps): scripts/verify-tarball.mjs uses `npm pack --dry-run --json`. The release-job dry-run uses the YAML `run:` shell directly (ubuntu-latest), so `npm publish --dry-run` is invoked as a plain shell command, not the process.execPath workaround.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions release job → npm registry | the release job holds OIDC id-token publish authority; everything it does crosses into the immutable public registry |
| GitHub-restored cache → privileged release job | a poisoned package-manager cache could inject malicious deps into a credentialed build (the reason D-09 removes the cache) |
| pushed tag ref → published artifact version | GITHUB_REF_NAME is attacker/operator-controllable; without the preflight a wrong tag publishes the wrong version |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-20-04 | Elevation of Privilege | OIDC id-token in release job | mitigate | Keep `permissions:` minimal (`contents: read`, `id-token: write` only); no stored long-lived token (D-01). Pure-OIDC short-lived token, retryable on failure. |
| T-20-05 | Tampering | package-manager cache in privileged release job | mitigate | D-09 removes `cache: 'npm'` from the release job (cache-poisoning hardening, actions/setup-node #1358); ci.yml keeps its cache (unprivileged). |
| T-20-06 | Tampering / Spoofing | wrong-commit or version-mismatch tag → publishes wrong artifact | mitigate | D-07 preflight asserts GITHUB_REF_NAME == v<mcp version> and mcp==cli before any publish; fails the job loudly via ::error:: + exit 1. |
| T-20-07 | Denial of Service (partial irreversible release) | independent non-transactional mcp/cli publish steps | mitigate | D-08 dry-run-both gate runs before either real publish, surfacing per-package PACK/MANIFEST errors while the registry is still untouched. Residual: the dry-run does NOT prove OIDC auth (M1), and npm has no atomic multi-publish — the auth-failure-after-mcp-published partial state is handled by the 20-05 recovery matrix (H2), not here. Accept the shrunk-but-nonzero window. |
| T-20-08 | Repudiation | published artifact provenance | mitigate | `--provenance` retained on both publishes; Node 22 (npm ≥11.5.1) makes OIDC provenance attestation work. Verified on npmjs.com in Plan 20-06 (SC3). |
