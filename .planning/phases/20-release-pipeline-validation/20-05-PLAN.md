<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| local tag ref → GitHub v* trigger → release.yml | pushing the tag is the single irreversible action; everything downstream publishes to the immutable registry |
| release job OIDC token → npm publish | short-lived id-token authorizes the publish; the trusted-publisher config (20-03, incl. Allowed actions: npm publish) must match or auth fails |
| two independent ordered publish HTTP transactions → registry | non-transactional, mcp-then-cli; a mid-sequence failure yields an irreversible partial release with mcp LIVE |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-20-15 | Tampering | tagging a commit that has not passed CI | mitigate | D-10 STEP A blocks tag creation until the bump commit's own ci.yml run is green on all 3 OSes (headSha-matched). |
| T-20-16 | Spoofing | tag content not matching intended version/metadata | mitigate | D-10 STEP C `git show v3.0.1:packages/{mcp,cli}/package.json` verifies version 3.0.1 + repository present BEFORE the push; wrong content → delete tag, restart. |
| T-20-17 | Denial of Service (irreversible partial publish) | independent ordered mcp-then-cli publish steps | mitigate | THREE layers: (1) D-08 dry-run-both (in release.yml) surfaces pack/manifest errors before either publish (but NOT auth — M1); (2) M3 pre-tag registry-state matrix asserts BOTH versions absent before the irreversible tag push; (3) the H2 recovery matrix (Task 3) handles a real partial publish correctly — never republish a live 3.0.1, publish only the missing cli, or bump BOTH to 3.0.2 for content changes (preserving D-07's matched-version invariant). Residual accepted: npm has no atomic multi-publish and versions are immutable — the window is shrunk and recovery is deterministic, not eliminated. |
| T-20-18 | Elevation of Privilege | OIDC id-token misuse / auth failure | mitigate | Pure-OIDC short-lived token (D-01); minimal permissions (20-02). Auth failure does NOT guarantee "nothing published" (ordered publish — M2/H2); the token fallback is UNWIRED and requires a release.yml env patch (M2), used only as the emergency path with the H2 recovery matrix. |
| T-20-19 | Repudiation | published artifact lacks provenance | mitigate | `--provenance` on both publishes + Node 22 (npm ≥11.5.1); provenance badge verified on npmjs.com in Plan 20-06 (SC3). |
</threat_model>
