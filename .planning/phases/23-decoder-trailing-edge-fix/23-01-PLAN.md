<!-- Confirmed at plan time via real-fs probes (node) + cross-model stress test (Codex gpt-5.5, 2026-06-30). -->
- Single trailing special `Trailing&/sub` encodes to `...-Trailing--sub`. The boundary needs `encodedName + '--'` (= "Trailing--"); the existing `encodedName + '-'` does NOT match. The additive `--` branch fixes it.
- D-01 casing subtlety CONFIRMED: for a lowercase-drive hash `c--tmp-Trailing--sub`, decode() produces `C:\tmp\Trailing&\sub` (drive upper-cased); a STRICT `encode(candidate) === hash` returns FALSE (false-reject → would regress L-03); the CASE-INSENSITIVE compare returns TRUE. Use `.toLowerCase()` on both sides.
- D-01 spurious rejection CONFIRMED: for hash `C--tmp-Foo--bar`, candidate `C:\tmp\Foo\bar` re-encodes to `C--tmp-Foo-bar` (single hyphen) ≠ hash → correctly rejected; candidate `C:\tmp\Foo&\bar` re-encodes to `C--tmp-Foo--bar` = hash → correctly accepted.
- Codex guard CONFIRMED: re-encoding the full decodedPath reproduces the EXACT raw hashDirName including the empty drive segment (`C--...`). Compare against the raw `hashDirName` argument, NOT a reconstructed-from-segments string.
- Cross-model NOTE (out of CORE-16 scope, handled in 23-02 as a documented boundary guard, NOT here): a component with TWO trailing specials (`Foo&&/sub` → `...-Foo---sub`, three hyphens) is matched by neither `-` nor `--`. CORE-16/SC1 scope is a SINGLE special char per position; do not generalize the branch in this plan (L-01 locks the exact `+ '--'` shape).
