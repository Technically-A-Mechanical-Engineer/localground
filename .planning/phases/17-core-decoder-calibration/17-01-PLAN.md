<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| filesystem listing → encode() regex | Decoder reads entry names via `fs.readdir` and feeds each name to `encode()`. Names originate from the local filesystem (user-controlled). |
| user-provided hashDirName → decode() | Public API consumer passes a string; consumer is local CLI/MCP tool, not network input. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-17-01 | Denial of Service | encode() regex `/[\\/: ,().'&\[\]+=%]/g` | accept | Flat character class with `/g` flag, no nesting, no alternation, no backreferences. Catastrophic backtracking (redos) is structurally impossible — character classes are linear. Verified by inspection. |
| T-17-02 | Tampering | decode() consuming attacker-controlled hashDirName | accept | Public API consumers (`@localground/cli audit`, `@localground/mcp localground_audit`) read hashDirName from `~/.claude/projects/` directory listings — a local, user-owned filesystem location. Not a network attack surface. The `segments.length < 2` guard at decode.ts:43 already rejects malformed input with `invalid_hash`. |
| T-17-03 | Information Disclosure | Round-trip tests creating fixture folders with special chars | accept | Tests use `os.tmpdir()` + `fs.mkdtemp` + `afterEach` cleanup. No PII in fixture names (`O'Brien`, `Rock & Roll`, etc. are deliberate plain-text test data). Fixtures are reaped before next test — no cross-test contamination. |
| T-17-04 | Spoofing | Forged path-hashes mimicking deleted folders | mitigate | Decoder relies on filesystem listing (`fs.readdir` at decode.ts:156). A forged hash that doesn't match any real subdirectory returns `no_candidates` — the safe failure mode. Phase 17 widens the encoder regex but does NOT loosen the decoder's verification step. |
</threat_model>
